Developer Guide

NoorTemplates architecture & extension guide

How the plugin is put together, how a WooCommerce product page actually gets rendered, and how to extend it safely — including exactly how the Pro companion plugin unlocks things.

Overview

NoorTemplates builds WooCommerce single-product pages out of Gutenberg blocks. A "Layout" is just a normal WordPress post (custom post type noortemplates_layout) whose block content wraps WooCommerce's own rendering — title, price, gallery, add-to-cart, reviews — each as its own block. Nothing is hand-coded per theme; a Layout is resolved fresh on every request and swapped in for the product template.

Three architectural decisions shape almost everything else in the codebase:

  • A Layout is content, not a template file. Store owners build it in the ordinary block editor, the same surface they already know. Layouts\Resolver decides which Layout post applies to which product at request time (see The Layout system) — there's no per-product PHP template to hand-edit.
  • Free/Pro is one codebase plus one filter, not two builds. Licensing\Gate is a plain static class whose is_pro()/has_feature() methods are pure filter lookups, defaulting to false. Nothing Pro-only is physically stripped from the free plugin — a separate companion plugin, NoorTemplates Pro, hooks those filters once a real license is active. See Licensing & the Pro gate.
  • Overriding the product template never breaks WooCommerce's own hooks. The override happens at the whole-template level (template_include), and the fallback template still fires every one of WooCommerce's structural actions (woocommerce_before_single_product, etc.) in their normal positions — so themes and other plugins hooked to those actions keep working around a Layout exactly as they would around WooCommerce's own template.

Everything under includes/ is PSR-4 autoloaded under the NoorTemplates\ namespace by includes/Autoloader.php, which maps NoorTemplates\Sub\ClassName to includes/Sub/ClassName.php and guards each require with is_readable() — safe to call class_exists() against a class whose file doesn't exist, with no fatal error.

Boot sequence

  1. plugins_loaded — noortemplates() resolves Plugin::instance(), which constructs the singleton (protected constructor, only reachable through the Singleton trait).
  2. WooCommerce presence is checked first. If class_exists( 'WooCommerce' ) is false, an admin notice is hooked and the constructor returns immediately — nothing else in the plugin boots at all without WooCommerce active.
  3. init_services() instantiates every service, in this order: Install, Blocks\Manager, Blocks\Interactivity, Blocks\Extensions, Blocks\Buy_Now, Patterns\Manager, Assets\Manager, Rest\Templates_Controller, Rest\Product_Reviews_Controller, Layouts\Post_Type, Layouts\Meta_Box, Layouts\Template_Override, Layouts\Duplicate, Layouts\Preview, Layouts\Split_Test — then, only if ( is_admin() ): Admin\Dashboard and Admin\Split_Tests_Page.
  4. Every service is a singleton constructed via its own ::instance(), each registering its own hooks in its constructor rather than being wired externally — there's no central "add_action" list to maintain in Plugin itself.
  5. do_action( 'noortemplates/loaded', $this ) fires once every service above exists — the hook a Pro or third-party add-on should use to register its own services, rather than hooking plugins_loaded directly and risking a load-order race.
  6. init handles textdomain, migrations, and registration that must wait for other plugins to have finished their own plugins_loaded: Plugin::load_textdomain(), Install::maybe_upgrade() (priority 5), Layouts\Post_Type::register(), Blocks\Manager::register_blocks(), Patterns\Manager::register_patterns().

The Layout system

A Layout is a post of type noortemplates_layout, edited in the ordinary block editor. Nothing renders it directly — on every product-page request, Layouts\Resolver::get_layout_id() decides which Layout (if any) applies, checking each of these in order and stopping at the first match:

  1. Preview override — a nonce-verified ?noortemplates_preview=ID query var from the Layout list's "Preview" row action. Works even for an unpublished Layout, gated by edit_post on that specific post.
  2. Active split test — if Split_Test::has_active_test( $product_id ), the visitor's bucketed variant's layout wins (see Split testing engine).
  3. Product-level choice — post meta _noortemplates_layout_id, set from the "Product Page Template" meta box on the product edit screen, if it points at a published Layout.
  4. Category-level default — term meta noortemplates_layout_id on the product's product_cat terms, first match wins.
  5. Site-wide default — option noortemplates_default_layout_id, set on the NoorTemplates Dashboard admin page.
  6. Nothing resolves — falls through to 0, and the product page renders exactly as WooCommerce/the active theme would without this plugin at all.

How the swap actually happens

Layouts\Template_Override hooks add_filter( 'template_include', ..., 100 ). Priority 100 is deliberate: WooCommerce's own class-wc-template-loader.php recomputes $template for product pages at its own priority, so this filter has to run after it to win. On is_product() requests where Resolver::get_layout_id() returns non-zero, $template is swapped for the plugin's own templates/single-product-layout.php.

That fallback template deliberately mirrors WooCommerce's real templates/single-product.php structure — same get_header('shop')/get_footer('shop'), same woocommerce_before_main_content / woocommerce_before_single_product / woocommerce_after_single_product / woocommerce_after_main_content / woocommerce_sidebar action firing, same $product global set via wc_get_product(). Where WooCommerce would normally render content-single-product.php, it instead does:

templates/single-product-layout.php
echo do_blocks( $layout->post_content );
ⓘ

Because every other WooCommerce structural hook still fires in its normal position, third-party plugins that hook woocommerce_before_single_product etc. (structured data, "recently viewed" trackers, upsell widgets) keep working identically around a NoorTemplates Layout — the override is a whole-template swap via template_include, never a the_content filter.

The "Product Page Template" meta box

Layouts\Meta_Box adds a side meta box to the Product edit screen: a dropdown of every published Layout (plus "— Default (theme/WooCommerce) —"), saved to _noortemplates_layout_id on save_post_product. It also adds a "Default Product Page Template" field to the product category add/edit screens, saved as term meta and requiring manage_product_terms. If Split Testing is licensed (see below), the same meta box grows a second section for configuring a test.

Split testing engine

Layouts\Split_Test lets a product's page A/B test between two Layouts. Every entry point funnels through one gate: has_active_test( $product_id ) requires Gate::has_feature('split_test') and _noortemplates_split_enabled === '1' and both variants pointing at published Layouts.

Bucketing (resolve_variant())

Reads cookie noortemplates_split_{product_id}. If it already holds a valid a/b value, that variant is reused (sticky bucketing across visits) with no re-logging. Otherwise a fresh roll (wp_rand(1,100) <= ratio ? 'b' : 'a') picks a variant, sets the cookie for 30 days via wc_setcookie(), and logs an impression. The result is memoized per product per request, since Resolver::get_layout_id() is called more than once per pageview.

Conversion logging

  • woocommerce_add_to_cart logs an add_to_cart metric for the visitor's existing cookie variant (a visitor with no cookie yet is not force-bucketed just to log this).
  • woocommerce_thankyou logs a purchase metric per order line item under an active test, guarded by the _noortemplates_split_logged order meta flag.

Counters live in a dedicated table, {$wpdb->prefix}noortemplates_split_stats (columns: product_id, variant, metric, count; unique key on the three), created via dbDelta() on activation. Split_Test_Stats::increment() is an atomic INSERT ... ON DUPLICATE KEY UPDATE count = count + 1.

⚠

There is no statistical-significance calculation and no date bucketing — the Split Tests admin page shows raw impression/add-to-cart/purchase counts and a plain conversion percentage. Treat it as a counter, not a significance test.

Declaring a winner

Admin\Split_Tests_Page ("NoorTemplates → Split Tests", manage_options) lists every product with an active test and its two variants' stats side by side. "Declare winner" on either variant copies that variant's layout into _noortemplates_layout_id, then deletes the B/enabled/ratio meta keys and resets the stats table for that product — ending the test and leaving only the winning Layout assigned normally. "Reset" clears counters without ending the test.

Templates & Patterns system

Bundled premade templates are plain JSON files under templates/library/, e.g. templates/library/bold-product-layout.json:

templates/library/bold-product-layout.json
{
    "title": "Bold Product Layout",
    "description": "…",
    "type": "layout",
    "category": "bold",
    "is_pro": true,
    "content": "<!-- wp:noortemplates/product-title /-->…"
}

type is required (layout or section); description, category, and is_pro are optional (category falls back to type, is_pro falls back to false). content always ships in full — this is a single codebase, nothing is stripped for a "Free" build. A thumbnail is auto-detected from {dir}/thumbnails/{filename}.{png,jpg,webp}, no JSON field needed.

Patterns\Manager scans every configured directory (get_pattern_dirs(), filterable via noortemplates/pattern_dirs — see below), caching the parsed result per-request and in a 1-day transient invalidated by an mtime fingerprint over every template + thumbnail file. Registration as native block patterns only happens where patterns are actually consulted (is_admin(), ajax, REST, or the noortemplates/patterns_are_needed filter) — skipped on plain front-end requests.

⚠

Any directory other than the plugin's own templates/library/ has every template in it force-set to is_pro = true, regardless of that JSON's own is_pro field. That folder only exists on a site because some add-on contributed it, so nothing in it is ever free by definition — see Licensing & the Pro gate.

Where a directory can come from

The base directory list is just array( NOORTEMPLATES_DIR . 'templates/library' ), run through apply_filters( 'noortemplates/pattern_dirs', $dirs ). NoorTemplates Pro is the one real example today — it appends its own templates/ folder (see Building a companion add-on).

Source aggregation (Templates\Repository)

The REST layer doesn't talk to Patterns\Manager directly — it goes through Templates\Repository, which merges every registered Source:

Local_Source

Wraps Patterns\Manager — the bundled (+ Pro add-on) JSON templates above. Implements Lazy_Source (fetches full content on demand).

Cloud_Source

Fetches a JSON array of templates from a remote URL (NOORTEMPLATES_CLOUD_URL or the noortemplates/cloud_api_url filter), transient-cached 12h on success / 5min on failure. Inactive if no URL is configured.

Sources are filterable via noortemplates/template_sources. On a name collision, the first source wins (bundled beats remote). The final merged, filterable-by-args collection is exposed via noortemplates/library_templates.

Licensing & the Pro gate

Licensing\Gate is the single source of truth for every free/pro decision in the codebase — a plain static utility, not a singleton, because every check is a pure filter lookup with nothing to instantiate:

includes/Licensing/Gate.php
class Gate {

    public static function is_pro() {
        return (bool) apply_filters( 'noortemplates_is_pro', false );
    }

    public static function has_feature( $feature ) {
        return (bool) apply_filters( "noortemplates_feature_{$feature}", self::is_pro() );
    }

    public static function feature_keys() {
        return array(
            'split_test',            // A/B split testing between two Product Layouts.
            'hero_video_background', // Video background on the Hero block.
        );
    }
}

is_pro() defaults false — the real, permanent Free behavior. Nothing Pro-only is physically removed from this codebase; every gate is a filter that a separate add-on can flip. has_feature($key) defaults to the global flag but lets a future tiered license override one feature independently, without changing any call site.

How NoorTemplates Pro unlocks things

NoorTemplates Pro (a separate, sibling plugin, Requires Plugins: noortemplates) never touches this plugin's internals directly — it only hooks the two extension points built for this purpose:

noortemplates-pro/includes/Plugin.php
class Plugin {

    public static function init() {
        add_filter( 'noortemplates_is_pro', array( __CLASS__, 'is_pro' ) );
        add_filter( 'noortemplates/pattern_dirs', array( __CLASS__, 'add_template_dir' ) );
    }

    public static function is_pro() {
        return function_exists( 'nt_pro_fs' ) && nt_pro_fs()->can_use_premium_code();
    }

    public static function add_template_dir( $dirs ) {
        $dirs[] = NOORTEMPLATES_PRO_DIR . 'templates';
        return $dirs;
    }
}

nt_pro_fs() is the vendored Freemius SDK's accessor (initialized in noortemplates-pro/includes/freemius-bootstrap.php against the plugin's own Freemius product). can_use_premium_code() is Freemius's own check for "this site has a valid license or active trial." The moment that's true, Gate::is_pro() flips to true everywhere in the free plugin — cascading through has_feature() to unlock Split Testing and the Hero video background, and through Rest\Templates_Controller to start serving Pro-tier template content.

ⓘ

This mirrors the sibling plugin NoorQuiz's own Core\Licensing design: the free plugin only ever exposes filter points, and a Pro add-on's entire job is hooking them with a real license check — nothing in the free codebase needs to know Freemius (or any licensing vendor) exists.

⚠

ProControl (the editor-side "PRO" badge component, see the Hero block's video background control) is UX only. It dims/labels a control in the block editor, but a determined user could still set a gated attribute directly — via the REST API, an imported JSON template, or hand-edited block markup. Every real gate has to be enforced again at the point the value is actually served or rendered: Rest\Templates_Controller::get_template() refuses Pro content server-side regardless of what the editor UI allowed; Split_Test::has_active_test() is the single choke-point every other Split Test code path calls through. A block that only checks Gate in its own edit.js and nowhere else is not actually gated.

Worked example: locking a feature

Split Testing is the clearest real example in this codebase of a feature gated correctly — at every point it's actually reachable, not just in one obvious spot. Three steps, in order.

1. Register the feature key

Every gated feature gets one entry in Gate::feature_keys() — purely a registry/documentation list (nothing reads it to enforce anything), so admin UI, docs, and other developers can see at a glance what's gateable:

includes/Licensing/Gate.php
public static function feature_keys() {
    return array(
        'split_test',            // A/B split testing between two Product Layouts.
        'hero_video_background', // Video background on the Hero block.
    );
}

This is the only step that's just bookkeeping. The next two are where the feature actually becomes locked.

2. Gate the one real choke point — server-side, not the UI

Find the single place in PHP where the feature's behavior actually happens, and check Gate::has_feature() there. For Split Testing, that's Split_Test::has_active_test() — every other Split Test code path (bucketing visitors, logging conversions, the results admin page) calls through this one method, so gating it here is enough to disable the feature's real effect everywhere at once:

includes/Layouts/Split_Test.php
public function has_active_test( $product_id ) {
    if ( ! Gate::has_feature( 'split_test' ) ) {
        return false;
    }

    if ( '1' !== get_post_meta( $product_id, self::ENABLED_META_KEY, true ) ) {
        return false;
    }

    $variant_a = (int) get_post_meta( $product_id, Resolver::PRODUCT_META_KEY, true );
    $variant_b = (int) get_post_meta( $product_id, self::VARIANT_B_META_KEY, true );

    return $this->is_published_layout( $variant_a ) && $this->is_published_layout( $variant_b );
}
ⓘ

Do this step first, before touching any UI. If the real behavior is already refused server-side when unlicensed, a half-finished or even entirely missing admin UI can never accidentally leak the feature — the worst case is a confusing UI, not a bypassed gate.

3. Gate the UI that configures/uses it

Once the real gate exists, make the *admin experience* match it — show what's possible as an upsell rather than just hiding it. The "Product Page Template" meta box renders one of two branches depending on Gate::has_feature('split_test'): the real controls, or a disabled checkbox with a "Pro" badge and an upgrade link:

includes/Layouts/Meta_Box.php
<?php if ( Gate::has_feature( 'split_test' ) ) : ?>
    <!-- the real checkbox, Variant B picker, traffic-split field -->
<?php else : ?>
    <p>
        <label>
            <input type="checkbox" disabled="disabled" />
            Split test against a second layout
            <span style="...badge styles...">Pro</span>
        </label>
    </p>
    <p class="description">
        Split testing requires NoorTemplates Pro.
        <a href="<?php echo esc_url( NOORTEMPLATES_CHECKOUT_URL ); ?>">Upgrade to Pro</a>
    </p>
<?php endif; ?>

The save handler applies the same check — skipping the save entirely when unlicensed, rather than treating "fields were disabled so nothing was posted" as "the user unchecked it":

includes/Layouts/Meta_Box.php
// Without the split_test feature the fields are rendered disabled
// (so nothing is posted for them) — skip entirely rather than
// treat "not posted" as "user unchecked it", which would silently
// wipe an existing split-test configuration just because the
// license lapsed before someone next saved the product.
if ( ! Gate::has_feature( 'split_test' ) ) {
    return;
}
⚠

That last detail matters as much as the gate itself: a lapsed license must never silently delete a merchant's existing configuration. Locking a feature means blocking new use of it, not destroying what was already there.

Applying this to a JS-only (block editor) feature

The Hero video background follows the identical three steps, just with an editor-side UI instead of a PHP admin screen: hero_video_background is registered in feature_keys(), the video-upload control is wrapped in <ProControl feature="hero_video_background"> (step 3's equivalent), and — because Hero is a static block with no PHP render step — there is currently no step 2 for it. That gap is called out explicitly in Conventions & gotchas: it's editor-UX-only until a real server-side check is added (a render_block filter, most likely), which is exactly the kind of gap step 2 exists to close for any feature that does have a real PHP-reachable choke point.

Extending NoorTemplates

Extension points at a glance

Every extension point follows the same shape: a filter that returns an array (or a boolean), into which your own plugin pushes one more entry. Nothing requires editing NoorTemplates' own files.

New blockfilter

Add a directory containing a block.json to noortemplates/block_dirs. Blocks\Manager auto-discovers and registers it the same way it registers the 29 built-in blocks.

New bundled templatefilter

Add your own directory to noortemplates/pattern_dirs, matching the JSON shape in Templates & Patterns system. Every template found there is automatically treated as Pro-tier.

New template sourcefilter + class

Implement Templates\Source (or Lazy_Source for on-demand content) and add it via noortemplates/template_sources — same shape as the bundled Cloud_Source.

Gate a new Pro featurefilter

Add a key to Gate::feature_keys(), call Gate::has_feature('your_key') at the one real point the feature is used (server-side, not just in editor JS), and hook noortemplates_feature_your_key (or just noortemplates_is_pro) from your add-on.

Disable a blockfilter

Add its name to noortemplates/disabled_blocks — the same filter the Dashboard's block-toggle settings write to.

Full companion add-onfilter + plugin

Hook noortemplates/loaded from a separate plugin to register your own services once NoorTemplates has fully booted — see below.

Building a companion add-on

NoorTemplates Pro is a real, working example of this pattern — not a hypothetical. It's a separate plugin folder, noortemplates-pro, that does nothing but vendor the Freemius SDK and hook two filters.

1. Plugin skeleton

A companion plugin, not a modification of NoorTemplates itself — Requires Plugins lets WordPress core handle the dependency check (deactivation/warning if the free plugin is missing) with no custom code.

noortemplates-pro.php
<?php
/**
 * Plugin Name:       NoorTemplates Pro
 * Description:       Unlocks Pro templates and Pro features for NoorTemplates.
 * Requires Plugins:  noortemplates
 */

defined( 'ABSPATH' ) || exit;

define( 'NOORTEMPLATES_PRO_DIR', plugin_dir_path( __FILE__ ) );

require_once NOORTEMPLATES_PRO_DIR . 'includes/freemius-bootstrap.php';

require_once NOORTEMPLATES_PRO_DIR . 'includes/Autoloader.php';
\NoorTemplatesPro\Autoloader::register();

add_action( 'plugins_loaded', array( '\NoorTemplatesPro\Plugin', 'init' ) );

2. Hook the two filters

Shown in full in Licensing & the Pro gate above — noortemplates_is_pro to unlock every Gate::has_feature() check at once, and noortemplates/pattern_dirs to contribute the add-on's own templates/ folder so real Pro template content physically lives in the Pro plugin, never in the free one.

3. A more targeted alternative: hook a single feature

If an add-on should unlock one feature rather than everything, hook the feature-specific filter instead of the global one:

PHP
add_filter( 'noortemplates_feature_split_test', function () {
    return my_addon_has_a_valid_license();
} );

This leaves Gate::is_pro() itself untouched (so templates and every other feature stay gated normally) while unlocking just Split Testing.

4. Registering your own service once NoorTemplates has booted

For anything beyond the two filters above — a new admin page, a new block, a background job — hook noortemplates/loaded rather than plugins_loaded, so every core service is guaranteed to already exist:

PHP
add_action( 'noortemplates/loaded', function ( \NoorTemplates\Plugin $plugin ) {
    \My_Addon\Service::instance();
} );

Other extension recipes

A new block

PHP
add_filter( 'noortemplates/block_dirs', function ( array $dirs ) {
    $dirs[] = __DIR__ . '/build/blocks/my-block';
    return $dirs;
} );

The directory needs a block.json, same as any of the 29 built-in blocks under src/blocks/ — Blocks\Manager doesn't care whether the directory belongs to this plugin or an add-on.

A new bundled template

PHP
add_filter( 'noortemplates/pattern_dirs', function ( array $dirs ) {
    $dirs[] = __DIR__ . '/templates';
    return $dirs;
} );

Drop .json files matching the schema in Templates & Patterns system into that directory, plus optional thumbnails in a thumbnails/ subfolder. Remember: every template found in an add-on-contributed directory is automatically Pro-tier, regardless of its own is_pro field.

A new remote template source

PHP
use NoorTemplates\Templates\Lazy_Source;

class MyRemoteSource implements Lazy_Source {
    public function get_id() {
        return 'my-remote-source';
    }

    public function get_templates() {
        // Return an array of template metadata (no content).
    }

    public function get_template( $name ) {
        // Return one template's full content, or null.
    }
}

add_filter( 'noortemplates/template_sources', function ( array $sources ) {
    $sources[] = new MyRemoteSource();
    return $sources;
} );

Working in this codebase

Development environment

You need a working WordPress site with WooCommerce active — NoorTemplates refuses to boot without it (see Boot sequence), so there's no way to work on it against a bare WordPress install.

If you're also working on NoorTemplates Pro alongside the free plugin, install it as a normal sibling plugin at wp-content/plugins/noortemplates-pro and activate both — it has no build step of its own (plain PHP + the vendored Freemius SDK), so there's nothing to npm install there.

Creating a new block

This walks through adding a block to this codebase directly (as opposed to registering an external block from a separate add-on plugin, which is a different recipe using the noortemplates/block_dirs filter instead of the steps below). Every block lives in its own folder under src/blocks/; Blocks\Manager::get_block_dirs() auto-discovers every folder under the built build/blocks/* matching one via glob() and calls register_block_type( $dir ) on each — so a new block only needs the right files in the right shape, no manual registration call anywhere in PHP.

The minimum file set

The simplest real block in this codebase, Advanced Button (src/blocks/button/), is a good template to copy:

src/blocks/button/ ├── block.json # name, attributes, supports — the single source of truth WP reads at registration ├── index.js # registerBlockType() — wires block.json + Edit + save together ├── edit.js # the editor-side React component ├── save.js # static markup serialized into post_content (omit entirely for a dynamic block) ├── style.scss # front-end + editor styles └── editor.scss # editor-only styles
src/blocks/button/index.js
import { registerBlockType } from '@wordpress/blocks';

import metadata from './block.json';
import Edit from './edit';
import save from './save';
import './style.scss';
import './editor.scss';

registerBlockType( metadata.name, {
    edit: Edit,
    save,
} );
  1. Create the folder under src/blocks/your-block/.
  2. Write block.json — set "name": "noortemplates/your-block", "category": "noortemplates" (the plugin's own inserter category), and declare every attribute you need.
  3. Write edit.js — the Inspector controls and in-canvas editing UI. If the block should get the plugin's own icon picker, unique-id helper, or Pro upsell badge, reuse the shared components in src/utils/ rather than rebuilding them (icon-picker.js, use-unique-id.js, pro-control.js).
  4. Choose static or dynamic rendering: a static block ships a save.js whose output is serialized straight into post_content (fine for pure layout/content blocks with nothing to compute at render time). A dynamic block (any block that needs live WooCommerce data — see every Product block in Codebase map) has no save.js and instead declares "render": "file:./render.php" in block.json, which webpack copies alongside the built JS (--webpack-copy-php in the build script — see Build tooling & shipping).
  5. Run npm run start (or build for a one-off) — the block appears in the inserter under the "NoorTemplates" category as soon as build/blocks/your-block/block.json exists and the plugin's init hook re-runs (reload wp-admin).
ⓘ

Every NoorTemplates block automatically gains the shared Visibility/Animation Inspector panel and front-end behavior from src/extensions/ — you don't opt in explicitly, it's applied to any block whose name starts with noortemplates/. If your new block is dynamic, double-check its render.php actually applies the visibility classes/animation data-attributes itself; the extension's save-props filter only fires for static blocks' save.js output.

Working in this codebase

Conventions & gotchas

  • Every service is a singleton via the shared Singleton trait (protected constructor, cloning disabled) — access an already-booted service through its own ::instance(), never new.
  • Meta and option keys are always class constants, never inline string literals — e.g. Resolver::PRODUCT_META_KEY, Split_Test::VARIANT_B_META_KEY. Reference the constant, not the raw string, so a rename only touches one place.
  • Split_Test::resolve_variant() is memoized per product per request — Resolver::get_layout_id() is called more than once per pageview, and re-rolling the bucket on the second call would break sticky bucketing within a single request.
  • An add-on's contributed template directory is always Pro-tier, regardless of the JSON's own is_pro field — see the callout in Templates & Patterns system. Don't rely on setting is_pro: false in an add-on's own template to make it free; it won't work, by design.
  • ProControl is editor-UX only, never the real gate. Always pair it with a matching Gate::is_pro()/Gate::has_feature() check at the point a value is actually rendered or served — see the callout in Licensing & the Pro gate.
  • No the_content filtering anywhere. The single-product override is a whole-template swap via template_include at priority 100 — keep it that way if you're touching Layouts\Template_Override, since filtering content instead would break the WooCommerce structural hooks other plugins depend on.
  • Blocks\WP_Icon_Registry is generated, not hand-written — it mirrors the @wordpress/icons package via scripts/generate-icon-registry.js. Regenerate it rather than editing it directly if the icon set changes.

Build tooling & shipping

CommandWhat it does
npm run startWatches src/, rebuilding build/ on change.
npm run buildOne-off production build of every block and editor script.
npm run lint:js / lint:csswp-scripts linting for JS and SCSS.

Built with @wordpress/scripts (--experimental-modules --webpack-copy-php, since several blocks ship a view.js as a script module and a render.php alongside their JS). NoorTemplates Pro has no build step of its own today — it's plain PHP plus the vendored Freemius SDK, so shipping it is just zipping the folder directly.

Shipping a release

  1. Bump the version in two places: the Version: header in noortemplates.php (which also sets the NOORTEMPLATES_VERSION constant), and Stable tag: in readme.txt. Unlike NoorQuiz's release script, nothing in this codebase cross-checks these automatically today — they can silently drift, so bump both by hand every time.
  2. Run npm run release — builds build/ fresh, then zips exactly the paths listed in package.json's files array (currently noortemplates.php, readme.txt, includes, patterns, templates, build) via wp-scripts plugin-zip. src/ and node_modules/ are never included — only compiled output ships.
  3. Sanity-check readme.txt separately from the version bump — it's a real wp.org-format readme (Requires at least, Tested up to, Requires PHP, feature list), not a stub, so keep its description in sync with what actually shipped.
⚠

As of this writing, noortemplates.php's header version and readme.txt's Stable tag/package.json's version are already out of sync (1.1.0 vs 1.0.0) — a concrete example of exactly the drift the step above warns about. Worth reconciling before the next release.
